I spent a good chunk of time this week chasing a ghost in a headless browser CI environment — a micro-frontend sub-application that simply refused to mount, with no errors, no console output, nothing. The symptom was deceptively clean: all HTTP requests returned 200, the container element existed in the DOM, and yet it stayed empty across every polling interval.

The diagnosis required building a layered instrumentation approach: injecting scripts to observe DOM mutations, hook history APIs, capture CSP violations, and poll for specific global variables. Each layer ruled out a hypothesis. No CSP errors. No routing events. No JavaScript exceptions escaping to the global scope. The micro-frontend framework's internal error handling was silently swallowing failures, which meant the real cause left no trace at the application level.

The actual root cause turned out to be a network namespace mismatch between the Node.js test runner process and the Chromium browser process inside the same CI container. Node.js inherited the machine's internal proxy and routing configuration, so it could reach the internal CDN. Chromium, running in its own network context, could not. Every asset fetch silently aborted, leaving the framework with empty script content and no lifecycle functions to register — and therefore nothing to mount.

The fix was to intercept all CDN requests at the Playwright router level and re-issue them from the Node.js side, then fulfill the browser's request with the response. This effectively proxies internal network access through the test runner process, bypassing Chromium's network stack entirely. It's a pattern worth keeping in mind whenever headless browser tests run in containerized CI: the browser and the test process may not share the same network view, and silent fetch failures can masquerade as application bugs.
