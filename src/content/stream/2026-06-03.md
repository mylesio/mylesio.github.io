Today I burned a lot of time iterating through four rejected solutions for what seemed like a simple layout bug: a pagination bar overlapping the last row of a table inside a chart component. The root cause turned out to be two separate issues tangled together — a prop not being forwarded down the component tree, and a non-scoped CSS rule in a parent component silently overriding styles globally. Neither was obvious from the surface symptom.

The fix that finally stuck was the least invasive one: compute an adjusted max-height inside the component itself, subtracting the paginator's height only when pagination is active, and leave the outer container's constraint completely untouched. Every earlier attempt tried to move or resize the outer max-height, which broke the layout contract the reviewer cared about most.

The real lesson isn't about CSS specificity — it's about respecting implicit constraints. When a layout has been carefully tuned by someone else, the safest fix absorbs the new requirement internally rather than reshaping the surrounding structure. And before touching any shared style, it's worth grepping for every other place that selector appears; a non-scoped rule in a distant file can be the invisible hand overriding your careful local fix.
