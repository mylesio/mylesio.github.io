Reading ECharts source code to solve a performance problem is a surprisingly effective debugging technique. When documentation is sparse and the official handbook lacks a dedicated performance section, dropping into the actual implementation reveals optimization hooks that are otherwise invisible. In this case, tracing through `dispatchHighDownActually` and `_updateContentNotChangedOnAxis` directly pointed to four actionable config changes — `triggerEmphasis: false`, `transitionDuration: 0`, `silent: true` on dispatched actions, and LTTB sampling — that together significantly reduce cross-chart linkage overhead. The real lesson here isn't just about those specific flags, but about the value of treating library source as a first-class debugging tool rather than a last resort.

A separate but equally important insight came from the Vue 2 reactivity trap. Storing a large third-party object (like an ECharts instance) as a plain class field causes Vue 2's `initData` to deeply observe it, triggering a cascade of `addDep` and `reactiveGetter` calls on every computed evaluation. The fix — a simple `_` prefix on the field name — leverages Vue 2's built-in `isReserved` check (`charCodeAt(0) === 0x5F`) to skip reactivity entirely. This is a clean, idiomatic pattern that's already used throughout many codebases, but it's easy to forget when you're focused on integration rather than storage semantics. The trace comparison was striking: `addDep` at 139ms and `reactiveGetter` at 86ms both vanished after the rename, confirming that sometimes the biggest performance wins come not from optimizing the hot path, but from stopping a whole subsystem from firing at all.
