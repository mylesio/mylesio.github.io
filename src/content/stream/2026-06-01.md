Today's most impactful lesson came from debugging the dashboard switching lag — a problem that looked like a simple rendering issue but turned out to be a silent performance bomb in Vue 2's reactivity system. The root cause was a `chartTitleMap` object exposed via provide/inject. Because it was a reactive object, every `small-chart` component that injected it automatically subscribed its watcher to each key's dependency. With N charts and M keys, that meant O(N×M) subscriptions. When switching dashboards, destroying N components required each one to splice itself out of M dep arrays — an O(N×M) operation that blew up to 64 seconds of flushCallbacks.

The fix was deceptively simple: replace the reactive object with `Object.create(null)`. Vue 2 doesn't observe objects created this way (they lack `__proto__`, so `isPlainObject` returns false and no getter/setter recursion happens). Since the data flow was write-then-read — not real-time reactive — there was no reason to pay the reactivity tax. This taught me a broader principle: whenever you're sharing data between components via provide/inject, ask whether it truly needs to be reactive. If not, a plain object is not just cleaner — it can be the difference between a smooth UI and a 64-second freeze.
