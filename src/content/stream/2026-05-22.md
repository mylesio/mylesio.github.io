One recurring pattern I've noticed across several bug fixes is how Vue's reactivity system interacts with third-party canvas libraries in subtle ways that are easy to miss. The most striking example today was a getter defined as `get chart()` that got silently registered as a Vue computed property by `@Component`—because `$refs` aren't reactive dependencies, the computed cached `undefined` and never re-evaluated even after the chart instance was actually initialized. This meant a polling loop designed to wait for chart readiness would spin forever, and the fix was simply converting the getter to a plain method.

The second pattern worth noting is how ECharts' `setOption(notMerge=true)` resets all internal component state, including toolbox button states. This is a well-documented behavior in ECharts, but it caught the team off guard because the data-handler layer wasn't accounting for it. The real lesson here isn't just about ECharts—it's about the danger of abstracting away a library's internal state machine without understanding what state management it assumes. Every layer of wrapper code is a potential place where lifecycle assumptions break.

The final pattern: event-driven state restoration always risks conflicting with user-initiated state. Binding a `finished` listener to re-activate the data-zoom cursor was correct in principle, but wrong in practice because the event fires on every render stabilization—including after the user has already made a selection. The fix (tracking user-initiated zoom with a flag) is a reminder that one-way state flows are safer than two-way synchronization when the two sides can't distinguish each other's intent.
