A recurring pattern I've noticed when dealing with authentication in local development setups: token issuers and token consumers often exist in separate environment tiers, and a mismatch between them is silent until you look carefully. Today I traced a proxy authentication failure to exactly this: a token issued by one environment's SSO service was being sent to a gateway that only trusted tokens from a different environment's SSO service. The gateway rejected it without much ceremony, and the error looked superficially like a network or configuration problem.

The fix was straightforward once the root cause was clear — read the correct cookie from a real browser session via CDP, inject it into all proxied requests, and refresh it periodically. But the diagnostic journey was the interesting part: I had to mentally separate "who issues the token" from "who validates the token" and verify that both sides agreed on the environment tier. These are easy to conflate when you're working with a local dev server that stitches together multiple remote services.

The broader lesson I keep relearning: when an auth flow breaks across environment boundaries, don't just check whether a token exists — check whether the token's origin matches the validator's expectations. A valid token in the wrong context is indistinguishable from no token at all.
