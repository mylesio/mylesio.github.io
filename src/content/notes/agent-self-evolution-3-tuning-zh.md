## 关键重构

微调改变的是 P(output|input)——模型在给定输入时产出特定输出的永久倾向。

Context 注入改变的是 P(output|input, context)——模型在给定相关经验的情况下的*条件性*倾向。

区别在于：
- 微调 = 肌肉记忆（永远在那里，不需要查找）
- Context 注入 = 开卷考试（需要翻到对的页面，但找到后效果一样）

人类需要内化知识是因为工作记忆只能容纳 ~7 个 item。LLM 有 128K–200K token 的 context window。**"开卷"非常大。** 在填满之前，"模型知道"和"模型被告知"没有功能差异。

```
┌──────────────────────────────────────────────────┐
│        什么时候 context 注入会失效？               │
├──────────────────────────────────────────────────┤
│                                                  │
│  1. 相关经验太多 (>20 个案例)                     │
│     → 检索精度成为瓶颈                            │
│                                                  │
│  2. 行为偏好太微妙                               │
│     ("语气"、"风格"、"品味")                      │
│     → 难以用示例表达                              │
│                                                  │
│  3. 对速度有极端要求                              │
│     → 检索延迟不可接受                            │
│                                                  │
│  在撞到这些墙之前：                               │
│  context 注入 ≈ 微调的效果                        │
│                                                  │
└──────────────────────────────────────────────────┘
```

## 路径 A：动态 Few-Shot 注入

### 核心思想

不检索*知识文档*，检索*决策案例*。

标准 RAG："这是关于 CSS 布局的文档" → Agent 读了 → 希望它能正确应用。

动态 Few-Shot："上次你遇到类似情况，这是发生了什么以及什么有效" → Agent 模式匹配 → 复制成功路径。

| 维度 | 标准 RAG | 决策案例检索 |
|------|---------|-------------|
| 检索的是 | 事实、文档、规范 | 过去的决策 + 结果 |
| 帮助回答 | "X 是什么？" | "X 该怎么办？" |
| 信号类型 | 知识性 | 行为性 |
| 索引键 | 主题相似度 | 情境相似度 |

### 实现流程

```
任务到达
    │
    ▼
提取任务签名：
  { type: "css_fix", signals: ["user_rejected_twice"], constraints: [...] }
    │
    ▼
向量检索 L3 判断力模式
  → 按情境相似度取 Top 2-3
    │
    ▼
注入 system prompt：
┌─────────────────────────────────────────────┐
│ ## 相关经验（自动检索）                       │
│                                             │
│ **类似情况**: 用户用结构性语言连续否决 CSS 方案│
│ ✅ 有效做法: 停止迭代，重新确认隐藏约束，     │
│    提出全新方案范式                           │
│ ❌ 无效做法: 在同一被否方案上调整参数          │
│    (浪费了 3 轮)                             │
│ 🔍 关键信号: "回滚/乱改了" = 方向错；        │
│    "差一点" = 继续迭代                       │
└─────────────────────────────────────────────┘
    │
    ▼
Agent 执行（行为被注入案例塑造）
```

### 设计决策

**为什么最多 2-3 个案例？** 超过 3 个，示例之间开始争夺注意力。模型可能混淆不同案例的建议。经验数据验证的收益递减点。

**Embedding 策略：** 对 L3 pattern 的 `trigger` + `context` 字段做向量化。查询时对当前任务的 type + 观察到的信号做向量化。用余弦相似度检索。

**过时管理：** 每个 pattern 有 `last_verified` 时间戳。60 天未验证的 pattern 检索分数打 0.5 折。120 天后归档（仍可搜索但不自动注入）。

## 路径 B：Prompt 进化（Mutator 模式）

### 核心思想

把 system prompt 当作**活文档**，基于证据进化。一个专门的"Mutator"流程定期根据积累的失败 episode 改写它。

```
┌─────────┐     ┌─────────┐     ┌─────────┐
│Prompt v8│────▶│ Agent   │────▶│ Traces  │
└─────────┘     │ 运行 N  │     │ (L1-L2) │
                │ 个任务  │     └────┬────┘
                └─────────┘          │
                                     │ 失败 episodes
                ┌─────────┐          │
                │Prompt v9│◀─────────┤
                └────┬────┘          │
                     │          ┌────▼────┐
                     │          │ Mutator │
                     │          │ (LLM)   │
                     │          └────┬────┘
                     │               │
                     ▼               ▼
              ┌────────────┐   ┌──────────┐
              │ Benchmark  │   │ 候选变更 │
              │ (通过/失败) │◀──│          │
              └────────────┘   └──────────┘
                     │
            通过: 部署 v9
            失败: 丢弃，保留 v8
```

### PRINCIPLES.md：可进化的单元

不重写整个 prompt，Mutator 操作的是一个结构化的 `PRINCIPLES.md`——一组行为规则，每条带元数据：

```yaml
- id: "P-007"
  trigger: "用户用结构性语言否决 ≥2 次"
  action: "停止当前方向，重新识别约束，换范式"
  source_episode: "ep-2026-06-03-pagination"
  verified_count: 3
  last_triggered: "2026-06-03"
  status: "active"  # active | candidate | retired
```

**生命周期：**

```
candidate ──(验证 3×)──▶ active ──(60 天未用)──▶ retired
    ▲                                              │
    │                                              │
    └──── (新失败 episode 触发创建) ◀──────────────┘
                                     (重新激活，如果
                                      又相关了)
```

### 进化节奏

- **每周最多 1 次变更**——防止级联变化导致无法诊断
- **每次变更 = git commit**，message："Add P-012 (source: ep-xxx, reason: ...)"
- **回滚粒度**：每次回退一条 principle，不是整个 prompt

## 路径 C：架构级路由

### 核心思想

不用一个通用 Agent 处理所有事情，而是把任务路由到**专门配置**——各自预加载了领域相关的 pattern、工具和约束。

这映射了波兰尼的观察：专业知识是领域特定的。国际象棋大师的直觉不能迁移到外科手术。Agent 的 CSS 修复判断力对 API 设计无益。

```
任务到达
    │
    ▼
┌───────────────────┐
│   分类器           │  (轻量级: 规则/embedding/小 LLM)
│   "这是什么类型    │
│    的任务？"       │
└───────┬───────────┘
        │
   ┌────┼────────┬──────────┐
   ▼    ▼        ▼          ▼
┌─────┐┌─────┐┌─────┐┌──────────┐
│CSS  ││API  ││调试  ││重构      │
│修复  ││设计  ││诊断  ││规划      │
│     ││     ││     ││          │
│原则: ││原则: ││原则: ││原则:     │
│1,7,9 ││2,4,8 ││3,5,6 ││4,10,11  │
│     ││     ││     ││          │
│工具: ││工具: ││工具: ││工具:     │
│edit, ││curl, ││grep,││ast,     │
│grep  ││test  ││log  ││diff     │
└─────┘└─────┘└─────┘└──────────┘
```

每个专家配置包含：
- **原则子集**（只保留相关的——噪音更少）
- **领域 few-shot 案例**（只用同领域的 episode）
- **受限工具集**（更少不相关的选项）
- **领域特定约束**（如 CSS 修复："先确认视觉约束"）

### 何时分裂

不要过早路由。分裂时机：

| 信号 | 门槛 |
|------|------|
| 不同任务类型有不同失败模式 | ≥ 3 种类型各有独立 L3 pattern |
| 领域间原则冲突 | 在 A 领域有效的原则在 B 领域有害 |
| L2 episode 量 | 每个候选领域 ≥ 20 个 episode |

**过度分裂的代价：** 维护负担上升 + 不属于任何专家的边界 case → 掉到退化的通用路径。

## 三条路径的组合

它们不互斥。在不同时间尺度运作，服务不同目的：

```
┌────────────────────────────────────────────────────────┐
│                   L3 判断力模式                          │
│                  (共享数据源)                            │
└───────┬──────────────────┬──────────────────┬──────────┘
        │                  │                  │
        ▼                  ▼                  ▼
 ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
 │  路径 A:     │  │  路径 B:     │  │  路径 C:     │
 │  Few-Shot    │  │  Prompt 进化 │  │  路由        │
 │              │  │              │  │              │
 │  "就在此刻， │  │  "从现在起， │  │  "对于这类   │
 │  对于这个    │  │  始终做 X"   │  │  工作，用    │
 │  任务"       │  │              │  │  配置 Y"    │
 │              │  │              │  │              │
 │  时效:       │  │  时效:       │  │  时效:       │
 │  每次请求    │  │  每周        │  │  每月        │
 └──────────────┘  └──────────────┘  └──────────────┘
```

**推荐采用顺序：**

1. **先用 B**（Prompt 进化）——门槛最低，第一天就能用人工 episode 审查
2. **加 A**（动态 Few-Shot）——当你有 50+ episodes 和向量存储后
3. **加 C**（路由）——当任务类型聚类在 L3 pattern 中清晰可辨后

## 更深的洞察

波兰尼的**寓居**（indwelling）概念描述了知识如何成为认知者的一部分——钢琴家不思考手指位置，他们"住在"钢琴里。

微调字面上实现寓居：知识进入权重，成为模型的一部分。

Context 注入*功能上*实现寓居：知识即时到达，模型的行为如同它一直知道一样。

实际限制不是能力——而是**检索精度**。Agent 自进化的真正工程挑战不是"如何学习"，而是"如何在对的时间记起对的事"。
