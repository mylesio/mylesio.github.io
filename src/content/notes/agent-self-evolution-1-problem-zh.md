## 悖论：聪明却失忆

一个 LLM Agent 可以进行多步推理、编排工具、写代码、调试自己的错误——全在一个 session 里完成。然后 session 结束，它忘记一切。

下次遇到同类问题时，它不记得什么有效、什么失败、用户的隐含偏好是什么。从零开始。

人类专家不是这样工作的。一个调试过 500 个 CSS 布局问题的资深工程师不会每次翻检查清单——他们在说出原因之前就*感觉*到哪个方向是错的。这种感觉就是**隐性知识**：通过实践积累，无法完全用规则表达。

迈克尔·波兰尼精确地概括了这一点：**"We can know more than we can tell."**（我们知道的比能说出来的多。）

Agent 的问题不是不够聪明，是失忆。

## 为什么"加更多规则"有天花板

行业的条件反射：Agent 犯了错 → 在 system prompt 里加一条规则。

```
Session 1:  Agent 破坏了用户的布局约束
            → 加规则："提方案前先确认约束条件"

Session 2:  Agent 问太多问题，用户不耐烦
            → 加规则："要果断，不要过度提问"

Session 3:  规则互相矛盾了
```

这种做法因四个结构性原因失败：

| 失效模式 | 机制 |
|---------|------|
| **注意力稀释** | Prompt 越长 → 模型对每条规则的关注度越低 → 遵循率下降 |
| **规则冲突** | "要主动" vs "要谨慎"——在不同场景下都对 |
| **空泛通胀** | "要认真"没有任何行为约束力——它是噪音 |
| **长尾覆盖** | 规则无法覆盖所有情境；真正的判断力应对的是新情况 |

**实证数据：** 在一个项目中，我们把 521 行的 system prompt 压缩到 209 行（−60%），跑同一套 benchmark。通过率：完全一样。60% 的规则什么都没做。

天花板不在于怎么写更好的规则，而在于规则能编码的东西的本质局限。

## 波兰尼的框架：显性 vs 隐性

```
┌─────────────────────────────────────────────────────┐
│                                                     │
│            可以写下来的                               │
│         ┌───────────────────────────┐              │
│         │       显性知识             │              │
│         │                           │              │
│         │  • "先检查约束条件"        │  ← 规则     │
│         │  • "先用 git blame"       │              │
│         │  • "最多重试 3 次"        │              │
│         └───────────────────────────┘              │
│                                                     │
│  ─ ─ ─ ─ ─ ─ ─ ─ 水面线 ─ ─ ─ ─ ─ ─ ─ ─ ─ ─     │
│                                                     │
│         ┌───────────────────────────────────┐      │
│         │          隐性知识                   │      │
│         │                                   │      │
│         │  • 感知用户即将失去耐心             │      │
│         │  • 在测试之前就知道哪种方案          │      │
│         │    "感觉不对"                      │      │
│         │  • 意识到连续两次失败 =             │      │
│         │    方向错了                        │      │
│         │  • 读取改变策略的上下文信号          │      │
│         │                                   │      │
│         └───────────────────────────────────┘      │
│                                                     │
└─────────────────────────────────────────────────────┘
```

波兰尼在熟练操作中识别了两层觉知：

- **焦点觉知**（focal awareness）：你正在有意关注的东西（任务本身）
- **辅助觉知**（subsidiary awareness）：你在无意识中整合的背景信号（用户语气、上下文模式、与过去问题的结构相似性）

专家的**从-到结构**（from-to structure）是这样工作的：感知辅助线索 → 无意识整合 → 得出焦点判断。他们往往说不清从感知到结论的路径。

对于 Agent，映射关系是：

| 波兰尼概念 | Agent 工程对应 |
|-----------|--------------|
| 辅助觉知 | trace 中的信号：被拒绝的方案、用户语气、上下文约束 |
| 焦点觉知 | 最终做出的决策 |
| 从-到结构 | 从"看到信号"到"做出判断"的模式 |
| 寓居（内化） | 模式成为 prompt / context / 路由逻辑的一部分 |

## ReAct Trace：被忽视的金矿

ReAct 框架（Thought → Action → Observation → 循环）为每个任务生成完整的决策轨迹。大多数团队只用这些 trace 来 debug。

他们丢掉了黄金。

```
┌─────────────────────────────────────────────┐
│  TRACE: CSS 布局修复任务                      │
├─────────────────────────────────────────────┤
│                                             │
│  Thought: "用户要修 max-height。             │
│   选项: (a) 移到子元素                       │  ← 决策点
│         (b) 增大数值                        │
│         (c) 用 overflow hidden              │
│   选 (b)——最简单的改动。"                    │
│                                             │
│  Action: 修改 max-height 170 → 210          │
│  Observation: 用户说"乱改了，回滚"           │
│                                             │
│  Thought: "用户强烈否决。                    │  ← 信号
│   '乱改了'说明方向错了，                      │
│   不是值不对。重新思考..."                    │
│                                             │
│  Thought: "如果约束是：                      │  ← 转折
│   外层 max-height 绝对不能动呢？             │
│   那我得在它内部解决。                        │
│   重新审视选项 (a): 传一个减去后的            │
│   高度给子组件。"                            │
│                                             │
│  Action: 添加 computed effectiveMaxHeight    │
│  Observation: 用户通过 ✓                     │
│                                             │
└─────────────────────────────────────────────┘
```

这里最有价值的数据不是"做了什么"——而是：

1. **考虑过但没选的方案**（第一次思考中的选项 a 和 c）
2. **什么信号触发了转折**（"乱改了" = 结构性否决，不是参数性的）
3. **第一次尝试和第二次之间的推理跳跃**（意识到一个隐藏约束）

这就是隐性知识的原材料。棋手复盘时关注的是*没走的棋*——因为那才是棋力的体现。

## 塑造一切的约束条件

大多数 Agent 开发者通过 API 调用模型，不碰权重。没有 LoRA，没有 DPO，没有 RLHF。

这意味着传统路径——"积累经验 → 内化到参数中"——不可用。

但有一个巨大的补偿优势：**context window 非常大**。

人类工作记忆约 ~7 个 item。所以人必须把知识内化为程序性记忆（肌肉记忆、直觉）——无法在主动回忆中保持所有经验。

LLM 有 128K–200K token 的 context window。这是一个巨大的外部记忆插槽，足以在推理时容纳数百条相关经验。

**重新定义问题：**

```
传统路径（微调）：
  经验 → 训练进权重 → 模型"自然就知道"

API 约束下（我们的现实）：
  经验 → 存入检索系统 → 运行时注入 → 模型"这次知道"
```

问题变成了：不是"如何让模型学会"，而是**"如何在每次推理时提供最相关的经验"**。

这直接引出了本系列后四篇的工程挑战：构建一个系统性地提取、结构化、存储和注入经验知识的管线。
