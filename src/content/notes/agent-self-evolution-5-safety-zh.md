## 核心风险

自进化系统最大的危险不是"不进化"——而是**往错误方向进化还不自知**。一个静默积累坏原则、膨胀 prompt、货物崇拜规则的 Agent，退化得如此缓慢，以至于没有任何一次 session 能暴露问题。

本篇覆盖安全网：确保系统只能朝正确方向移动（或原地不动），绝不倒退的机制。

## 蒸馏：安全的 Prompt 压缩

当你的 PRINCIPLES.md 增长到超出 token 预算时，需要压缩。问题是：如何移除内容而**不丢失有效原则**？

### 错误做法：凭直觉裁剪

"这条规则看起来多余，删了吧。" → 你刚好删掉了防止某类特定错误的那条规则。几周后它重现时你才知道。

### 正确做法：消融测试（Ablation Testing）

借鉴神经科学——要理解某个脑区的功能，观察移除它后什么会坏掉。

```
PRINCIPLES.md（当前，25 条原则）
         │
         │  对每个删除候选：
         ▼
┌─────────────────────────────────────────┐
│  1. 从 prompt 中删除 P-xxx             │
│  2. 跑 benchmark（同一组任务）           │
│  3. 与基线对比结果                       │
│                                         │
│  结果 A: 无变化 → P-xxx 是噪音          │
│  结果 B: 变差了 → P-xxx 是有效的         │
│  结果 C: 变好了 → P-xxx 本身有害         │
└─────────────────────────────────────────┘
```

**关键规则：一次只测一个删除。** 同时删除多条原则会让归因变得不可能。

### 四种蒸馏操作

| 操作 | 适用场景 | 示例 |
|------|---------|------|
| **删除** | 消融显示无影响 | "要仔细分析"（太模糊，无法影响行为） |
| **合并** | 多条说同一件事 | P-003 + P-007 + P-012 都关于"用户否决处理" → 合为一条 |
| **升级为 few-shot** | 规则太抽象 | "注意边界情况" → 替换为一对具体正反例 |
| **降级为检索** | 低频但验证有效 | 从常驻 prompt 移到 L3 案例库，检测到相关任务时才注入 |

**"降级为检索"是最重要的招。** 它让你缩小 prompt 而不丢信息。原则没消失——只是不再占据 prompt 空间，直到需要时才出现。

### 触发时机

当 prompt token 计数达到预算的 80% 时启动蒸馏。不要等满了——你需要为下一轮进化留出空间。

## 变异控制：四个门控

Mutator（Part 3）提议对 PRINCIPLES.md 的变更。不是每个提议都该被接受。四个门控过滤坏变异：

```
失败 Episode
      │
      ▼
┌──────────┐
│ Mutator  │ → 候选变更
└──────────┘
      │
      ▼
┌──────────┐
│  Gate 1  │  结构检查 — 形式是否正确？
│  (自动)  │
└────┬─────┘
     │ 通过
     ▼
┌──────────┐
│  Gate 2  │  实质检查 — 是否有证据支撑？
│  (自动)  │
└────┬─────┘
     │ 通过
     ▼
┌──────────┐
│  Gate 3  │  回归检查 — 是否破坏已有能力？
│  (自动)  │
└────┬─────┘
     │ 通过
     ▼
┌──────────┐
│  Gate 4  │  膨胀检查 — 是否超出 token 预算？
│  (人工)  │
└────┬─────┘
     │ 通过
     ▼
  接受 → 进入候选池
```

### Gate 1：结构检查

候选原则必须：
- 包含具体触发条件（"当 X 发生时…"）
- 包含具体动作（"…做 Y"）
- ≤ 3 句话
- 不是纯态度表述

```
❌ 拒绝: "修改 CSS 时要更加小心"
   原因: 无触发条件，无具体动作，纯态度

✅ 接受: "当用户用结构性语言（'回滚/乱改了'）否决方案 ≥2 次时，
   停止当前方向，重新识别约束后提出根本不同的方案"
   原因: 明确触发，具体动作，范围有界
```

### Gate 2：实质检查

- 必须引用至少一个来源 episode（"因为 ep-xxx 证明了…"）
- 必须解释反事实："如果这条原则在 ep-xxx 中存在，[某个决策点]的选择会从[坏]变为[好]"
- 不得与已有活跃原则矛盾

### Gate 3：回归检查

- 加入新原则后跑 benchmark
- 通过率不得下降
- 特别监控：过度谨慎（Agent 开始什么都问用户许可）

### Gate 4：膨胀检查

- 添加后 PRINCIPLES.md 总 token 数必须在预算内
- 若超预算：提议必须附带一个退役建议（"退役 P-xxx 因为…"）
- **达到容量时进一出一**

## 冷却期

最危险的变异模式：**一次失败 → 立刻加一条规则**。这是近因偏误陷阱。

```
Day 1: Agent 犯了错误 X
       → Mutator 提议 P-new: "永远检查 X"
       → 听起来合理，通过了 Gates 1-3...

Day 2-30: 错误 X 再没发生过
          (它是一次性的，不是模式)
          P-new 待在 prompt 里什么都没做
          ...除了消耗 token 和注意力

Day 31: P-new 和一条新原则冲突
        → 级联失效
```

### 解决方案：候选暂存区

新原则不直接进入活跃 prompt。它们进入**候选池**：

```
┌────────────────────────────────────────────┐
│  候选池                                    │
│                                            │
│  P-new (2026-06-03 提出)                   │
│  ├── 来源: ep-2026-06-03-pagination        │
│  ├── 验证次数: 0/3                         │
│  ├── 状态: 等待验证                        │
│  └── 过期: 2026-07-03 (30 天)             │
│                                            │
│  晋升标准:                                  │
│  - 相关场景出现 ≥3 次                       │
│  - 其中 ≥2 次，该原则本应改善决策           │
│                                            │
│  过期规则:                                  │
│  - 30 天内 <3 个相关场景                    │
│    → 自动丢弃                              │
└────────────────────────────────────────────┘
```

**冷却期确保只有反复出现的模式才成为原则。** 一次性错误留在 L2 episode 层（通过 few-shot 检索在相关时可用），但不污染始终存在的 prompt。

## 反模式目录

自进化系统常见的五种失效模式：

### 1. 态度通胀

```
❌ "要更加小心"
❌ "注意边界情况"
❌ "行动前仔细思考"
❌ "二次检查你的工作"
```

**为什么有害：** 零行为约束。模型已经在"尝试"仔细了——告诉它更努力尝试不会改变它做什么。这些积累起来稀释了对可操作原则的注意力。

**检测：** 任何不包含具体 trigger + action 的原则。自动正则检查：必须含有"当/如果" + 一个动词。

### 2. 过度泛化

```
❌ "修改前永远先问用户"
   (原因: 某次 Agent 改了用户不想改的东西)
```

**为什么有害：** 把 Agent 变成一个犹豫不决的问卷机器。用户给 Agent 工具就是为了让它自主行动。

**检测：** 包含"永远"或"绝不"且无范围限定触发器的原则。交叉检查：是否与任何"要果断"类指令冲突？

### 3. 因果幻觉

```
❌ "API 调用永远用 try-catch 包裹"
   (原因: 一次网络超时，跟错误处理毫无关系)
```

**为什么有害：** 原则针对的是症状而非原因。它在所有地方添加防御代码，无论实际失败模式是否相关。

**检测：** Gate 2 反事实检查——"try-catch 能防止原始故障吗？"如果不能 → 拒绝。

### 4. 冗余保险

```
❌ "检查文件是否存在。然后验证是否可读。然后确认非空。
    然后校验格式。然后解析。然后验证解析结果。"
```

**为什么有害：** 把 2 步操作变成 6 步偏执序列。大多数检查是冗余的（如果不可读，解析自然会失败）。

**检测：** 规定顺序验证步骤的原则，而单次操作就能暴露同样的错误。

### 5. 近因偏误（最危险的）

```
第1周: Agent 在任务类型 A 失败 → 加 P-new-A
第2周: Agent 在任务类型 B 失败 → 加 P-new-B
第3周: Agent 在任务类型 C 失败 → 加 P-new-C
...
第10周: Prompt 的 40% 是"单次失败的教训"
        而这些失败从未重现
```

**为什么有害：** Prompt 被一次性反应填满。信噪比崩溃。冷却期（上文）是主要防线。

**检测：** 跟踪每条原则的 `verified_count`。60 天后，verified_count < 2 的任何原则都是近因偏误的退役候选。

## 系统性退化：早期预警信号

即使所有门控到位，也要监控这些宏观指标：

| 信号 | 含义 | 行动 |
|------|------|------|
| Prompt 每周增长 >5% | 变异快于退役 | 严格执行进一出一 |
| Benchmark 方差增大 | Agent 行为变得不稳定 | 暂停变异，调查冲突 |
| 用户干预率反弹 | 近期变更弊大于利 | 回滚最近 2-3 次变异 |
| Fix-loop 次数上升 | Agent 在自我怀疑 | 检查过度谨慎的原则 |
| 首次正确率下降 | Agent 失去信心 | 可能存在原则冲突 |

## 版本控制与审计

对 PRINCIPLES.md 的每次变异应该：

1. **一次 git commit**，结构化 message：
   ```
   evolve(P-012): add constraint-detection principle
   
   Source: ep-2026-06-03-pagination
   Counterfactual: would have saved 3 fix-loops
   Ablation: benchmark 89% → 89% (no regression)
   Gate results: G1✓ G2✓ G3✓ G4✓
   ```

2. **关联评估数据**：哪次 benchmark 跑确认了它，前后指标是什么。

3. **可事后审查**：每季度审计进化日志。哪些原则真正触发了？哪些闲置了？这是防止长期漂移的"大扫除"。

## 回扣波兰尼

波兰尼观察到**传承**（tradition）——隐性知识从师傅到学徒的传递——需要特定结构：学徒必须在师傅的观察下练习，师傅在恰当时刻提供纠正性反馈。

自进化管线模拟了这个结构：
- **练习** = Agent 运行任务（产生 trace）
- **师傅的观察** = Evaluator 提取 episode
- **纠正性反馈** = L3 pattern 注入未来 session
- **安全网** = 确保"师傅"不教坏习惯

关键区别：人类师傅在传承中（定义上）不会错。自动化进化系统*可能*漂移。这就是门控、benchmark、冷却期和回滚机制存在的原因——它们扮演"实践共同体"的角色，验证"师傅的教导"是否正确。

## 系列总结

五篇文章构建了一个完整框架：

1. **问题**：Agent 是无状态的；规则有天花板；隐性知识是前沿
2. **管线**：L1 trace → L2 episode → L3 pattern → L4 注入
3. **方法**：动态 few-shot、prompt 进化、架构路由
4. **度量**：三层评估金字塔 + 判断力 benchmark
5. **安全**：消融蒸馏、四门控、冷却期、反模式

统一洞察：**无微调的 Agent 自进化是一个信息精炼问题，不是学习问题。** 你不是在教模型——你是在策展它看到的内容。工程挑战是最大化策展的相关性和精度，同时防止策展过程本身退化。

Context window 是你的画布。明智地填充它。
