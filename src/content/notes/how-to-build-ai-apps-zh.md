## 背景

### 行业背景

2024 年数据：AI 聊天机器人应用营收超 10 亿美元；用户在 AI 聊天 App 上消耗了 70 亿小时，同比增长 347%。大模型已经在市场层面带动了互联网行业 10 亿美元级别的发展。

随着模型进化为多模态（GPT-4o、Qwen2.5-Omni 等），C 端垂类应用更安全，但对内部工具来说，模型能力越强越好。

### 技术背景

从 Claude 3.5 Sonnet 发布开始，编程行业踏上了 AI 时代的朝圣之路。Anthropic 此后带来了混合推理模型、MCP 标准、用于快速应用文件编辑的 fast-apply 模型，以及 Cursor。旧的开发范式正在被主动瓦解。

如果 React 时代的工具给了我们上杠杆的机会，那 LLM 的出现是在升维——需要更广的技术广度，而不只是深度。

## 应用示例

通用 AI 聊天助手：Electron 外壳，React + Vite 前端，本地服务器，工具执行任务。数据可视化案例展示了这套架构的能力上限。

模式是：用户输入 → LLM 解读 → 调用工具 → 携带完整上下文返回结果。

## 底层实现拆解

### 3.1 MCP 原理

MCP（`modelcontextprotocol.io`）是一个协议，不是产品。最重要的原语是 `tool`——类比 function call，但在协议层面、与模型无关。

一个常见问题：OpenAI function call 和 MCP tool 有什么区别？关键区别：MCP 是一个协议，适配任何模型。function call 是模型特有的能力。MCP 在可移植性上胜出。

**请求循环：**

每次交互本质都是一次请求链：
1. 用户发送消息，系统提示词中包含 MCP server 配置
2. LLM 将工具使用包装为结构化标签（如 `<use_mcp_tool>`）
3. **客户端（不是 LLM）实际调用 MCP server**
4. 结果被追加到上下文，发回给 LLM
5. LLM 综合整个上下文返回响应

LLM 从不执行代码，只负责编排。

### 3.2 AI 应用架构

想象一个坐标轴，x 轴是功能范围，y 轴是上下文长度。工具执行时，结果在上下文中累积。这是当前所有 AI 应用的底层模式——RAG、浏览器操作、代码 Agent——都是上下文管理策略。

```
用户输入
    + 系统提示词
    + 工具定义
    + 工具执行结果（累积）
    = LLM 上下文
         → LLM 响应
```

架构到处都一样。差异在于你构建什么工具、如何管理上下文质量。

## 结语

MCP SDK 官方从一开始就支持了 Node.js 和 Python。但应用层在往 Node.js 靠拢。Atwood 定律还在提升含金量：任何能用 JavaScript 写的应用，最终都会用 JavaScript 写。

机会不在模型层——那是 Python 的领地。机会在工具和上下文编排层。那是前端工程师的主场。
