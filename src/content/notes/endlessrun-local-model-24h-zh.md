## 一句话描述

让端侧小模型（8B-26B）通过**任务分解 + Memory 压缩 + 原生 Tool Call**实现 24 小时持续运行复杂任务。

## 为什么值得做

- **零成本运行**：端侧模型不消耗 API 费用，算力免费
- **数据全在本地**：隐私安全，不上传云端
- **持续运行**：不受 API 限流、服务宕机影响
- **养成系 Profile**：越用越懂你，越用越高效

---

## 架构

```
端侧模型
    │ 输出带标记的结构化文本
    ▼
标记解析层（Runtime 核心）
    │
    ├── read_file    → 读取本地文件
    ├── run_script   → 执行本地脚本
    ├── save_memory  → 触发 memory 压缩
    ├── next_step    → 加载下一个子任务
    ├── checkpoint   → 保存当前状态快照
    └── abort        → 异常暂停，通知用户
    │
    ▼
执行结果塞回模型上下文 → 循环
```

模型本身不执行任何代码，只输出结构化调用指令。执行永远是外部 Runtime 的事。

## Gemma 4 原生 Tool Use 协议

六个专用控制 Token：

| Token 对 | 作用 |
|---|---|
| `<\|tool>` / `<tool\|>` | 定义可用工具 |
| `<\|tool_call>` / `<tool_call\|>` | 模型发起工具调用 |
| `<\|tool_response>` / `<tool_response\|>` | 将结果喂回模型 |

工作流示例：
```
1. 系统用 <|tool> token 声明工具
2. 模型需要工具时输出 <|tool_call>
3. Runtime 执行，将结果包装在 <|tool_response> 中
4. 模型继续生成
```

## 强约束机制

小模型容易跑偏，不能完全信任其自由输出。四个系统级约束：

**1. 强制 Memory 压缩**
```python
if output_tokens > 3000 or step_completed:
    # 强制触发 save_memory
    # 校验压缩质量
    # 通过 → 继续，失败 → 重新压缩
```

**2. 强制检查点**  
每个子任务完成后：保存进度 + 压缩后记忆 + 步骤摘要 + 下一步指令。支持断电/重启后从检查点恢复。

**3. 输出格式校验**  
解析 → 校验 → 执行。连续 3 次格式错误 → 触发 `abort`。

**4. 安全白名单**  
`rm -rf /` → 拒绝。`read_file /etc/passwd` → 拒绝。`python analyze.py` → 允许。安全边界必须硬编码，不能靠模型"自觉"。

## Memory 压缩方案

核心原理：用小模型自身做摘要压缩（小模型做信息压缩效果不差，这是其能力甜区）。

压缩提示词模板：
```
以下是上一步的完整输出。请提取关键结论，用结构化格式压缩到 200 tokens 以内。
必须保留：数据结论、决策结果、发现的问题
可以丢弃：推理过程、重复信息、客套话
```

双重校验机制：
```
Step N 完成
    ↓
模型 A：生成压缩摘要
模型 B（同一个模型，不同 prompt）：校验摘要是否遗漏关键信息
    ↓
通过 → Step N+1
失败 → 重新压缩
```

端侧模型不要钱，多跑几次校验完全没有成本压力。

## 上下文预算管理（8K 窗口示例）

```
总窗口：8192 tokens
├── Profile 注入（压缩后）：300 tokens  (3.7%)
├── 任务状态摘要：          500 tokens
├── 当前步骤指令：          200 tokens
├── 当前步骤输入（如文件）：  2000 tokens
├── 预留给模型输出：         4000 tokens
└── 安全余量：              1192 tokens
```

Google 官方建议：
> "对于长时间运行的 Agent 任务，建议把之前的思维过程提炼总结后注入回上下文。"

## 模型选型（Gemma 4，M3 Pro 24GB）

| 模型 | 内存（INT4） | 可行性 | 速度 |
|---|---|---|---|
| E2B（激活 23 亿） | ~3GB | ✅ 非常轻松 | ~30-40 tok/s |
| E4B（激活 45 亿） | ~5GB | ✅ 轻松 | ~20-30 tok/s |
| 26B MoE（激活 38 亿） | ~15GB | ✅ 能跑 | ~10-15 tok/s |
| 31B Dense | ~18GB | ⚠️ 勉强 | ~5-8 tok/s |

推荐策略：
- **开发调试**：E4B（8B），占用 6-8GB，留足内存给 IDE
- **正式跑任务**：26B MoE，关掉不必要的应用

## 与模型厂商的关系

| 维度 | 模型厂商 | EndlessRun |
|---|---|---|
| 方向 | 通用能力提升 | 个性化能力放大 |
| 关系 | 厂商提升地板 | 你提升天花板，互补而非竞争 |

三层自保：
1. **数据层锁定**：积累的"哪种模型 + 哪种 prompt 模式在你的场景最有效"的经验库，与具体用户绑定，厂商拿不走
2. **模型无关性**：不绑定任何一家模型，任何一家做好了你都受益
3. **占住用户侧**：你是用户的代理人，不是模型的附属品
