## 问题

多个 AI Agent 协同工作时，四个问题反复出现：

1. **角色边界不清** — Agent 互相踩脚
2. **授权链断裂** — 谁批准了这个破坏性操作？
3. **任务分解难追踪** — 这个子任务从哪来的？
4. **跨 Agent 质量控制缺失** — 每个 Agent 给自己批改作业

本文整合了构建多 Agent 治理协议的经验（灵感来自 Anthropic 的 Harness 工程博客，以及在真实代码库上跑 Agent 团队的实践）。

---

## 核心架构：Queen / Worker / Evaluator

| 角色 | 职责 | 关键约束 |
|------|------|---------|
| **Queen** | 编排、进度控制、质量门禁、授权决策 | 不应成为主要实现者 |
| **Worker** | 任务分解、实现、交付 | 无无限自主权；敏感操作需请求授权 |
| **Evaluator** | 独立评估 Worker 产出 | 必须调成"挑剔模式"+ few-shot 校准 |

**核心洞见：Evaluator 不能由 Generator 兼任。** 模型评估自己的输出时天然过于宽容。独立 Evaluator 能发现自检漏掉的逻辑 Bug——无论是主观任务（设计）还是客观任务（代码正确性）都是如此。

## 授权模型（L0–L3）

| 级别 | 触发条件 | 授权方 |
|------|---------|-------|
| **L0** | 读文件、分析、草稿、本地构建/测试 | 无需授权 |
| **L1** | 修改重要逻辑、启动有影响的服务 | Worker → Queen |
| **L2** | 大范围但可逆的修改 | Queen 可代批（须在报告中披露） |
| **L3** | 破坏性/不可逆操作、核心配置变更 | Queen + 人类双重确认 |

每条授权有唯一 ID。每个 Agent 有全局唯一标识符。禁止匿名 Agent。

## Context Reset 优于 Context Compaction

上下文接近上限时，模型会出现"Context Anxiety"——提前收尾、行为退化。这不是 token 不够的问题，而是模型的心理状态问题。

| 方法 | 机制 | 适用场景 |
|------|------|---------|
| **Compaction**（摘要压缩） | 保留连续性，就地缩减 | 短任务、模型能力足够 |
| **Context Reset**（清空重开） | 清空窗口 + 结构化交接 | 长任务、跨天任务、anxiety 明显时 |

**Compaction 不能根治 Context Anxiety。Context Reset 才能。**

交接产物（HANDOFF.md）必须包含：当前进度、已完成标准、剩余标准、已知阻塞、下一步行动、关键上下文（文件路径、踩坑点）。

## Sprint 合约

计划保持高层（用户故事级），避免 Planner 错误向下游传播。但高层 spec 和可验证实现之间存在 gap。

**Sprint 合约弥合这个 gap：**

1. Generator 提议："我打算实现 X，验证方式是 Y"
2. Evaluator review：确认验收标准
3. 双方对齐 → 开始开发

Agent 间通信通过文件进行（天然异步、可持久化、可审计）。

---

## Evaluator 必须用活的交互

| 验证方式 | 能发现的 | 发现不了的 |
|---------|---------|-----------|
| 静态截图 | 布局错误、颜色偏差 | 按钮存在但 API 没发出 |
| CDP 实际点击 | 以上 + 交互逻辑、API 路由 | 深层竞态 |
| Playwright 自动化 | 同 CDP，更系统化 | 同上边界 |

**静态截图看不到"按钮在那儿但点了没反应"。** Evaluator 必须实际走关键用户路径。

## Prompt 进化系统

自动优化 Agent Prompt 的闭环系统：

```
运行基线评估（当前 prompt 打分）
    ↓
生成变异（LLM 分析失败案例，提出修改）
    ↓
在 benchmark 套件上评估变异后的 prompt
    ↓
得分提升？→ 接受变异，更新最佳 prompt
得分不变/下降？→ 丢弃，尝试下一个变异
    ↓
循环直到预算耗尽或达到目标得分
```

关键约束：
- **同时只能跑一个进化实例**（否则共享状态污染）
- **每次完整 run 前必须跑 smoke test（22 项检查，30 秒）**
- **每个变异是结构化的规则修改**，不是随机编辑

实践结果：基线 79.2% → 最佳 87.5%，两个被接受的变异带来 +8.3pp 提升。

## Harness 瘦身原则

> 每个 harness 组件都是一个假设："模型做不到 X。"

模型升级后，这些假设需要定期重新验证。协议：

1. 一次只移除一个组件
2. 观察对最终输出质量的影响
3. 不要激进重构（分不清哪些是承重结构）
4. 每次大版本模型升级后做一轮 review

Sonnet 3.5 时代的必要脚手架，到下一代模型可能就变成了不必要的开销。

## 目标

将人类介入缩减到恰好三个触点：

1. **需求确认** — 确认 spec 一次
2. **异常升级** — Agent 重试 N 次仍失败
3. **最终验收** — 截图 + 流水线全绿，确认上线

其他一切由 Agent 团队自治处理。Queen 通过 Sprint 合约维护质量，Evaluator 提供独立验证。
